## The Architecture (`train/custom_transformer.py`)

We do not use generic models. We built a highly optimized, Decoder-Only Transformer scaled to ~2.92 Billion parameters specifically for B2B Cloud and Server Security.

**Key Mathematical Features:**
1. **Flash Attention 2:** Replaces standard attention to provide $O(N)$ memory scaling. Allows the model to ingest massive 8,192-token context windows (e.g., full Nginx logs or raw eBPF traces) without running out of VRAM.
2. **Rotary Position Embeddings (RoPE):** Allows perfect extrapolation across varying lengths of system logs.
3. **Grouped Query Attention (GQA):** Uses 32 query heads but only 8 key/value heads. This slashes memory bandwidth requirements, allowing the model to infer decisions in **100-300 milliseconds**.
4. **Gradient Checkpointing:** Saves VRAM during the backward pass so the 3B model can be trained on standard GPU clusters without OOM errors.

## The Training Pipeline

To build this model, follow these 5 phases:

### Phase 1: Data Ingestion
Before training, you must gather terabytes of raw cybersecurity data. The model needs to learn the "physics" of Linux servers and hacking tools.
- **Script:** `uv run python train/ingest_kali_linux_corpus.py`
- **What it does:** Generates corpuses of Kali Linux commands (e.g., `sqlmap`, `nmap`), bash privilege escalation commands (`cat /etc/shadow`), and raw eBPF syscalls.
- **Data Location:** All raw `.txt`, `.json`, and `.log` files must be placed in `train/data/raw_corpus/`.

### Phase 2: Custom Tokenizer Training
Standard AI tokenizers (like GPT-4's) are optimized for English and will destroy IP addresses, hex memory codes, and file paths.
- **Script:** `uv run python train/train_custom_tokenizer.py`
- **What it does:** Uses the HuggingFace `tokenizers` library to train a custom **Byte-Level BPE Tokenizer**. It scans the `raw_corpus` folder and permanently adds high-frequency hacker vocabulary (like `sys_enter_execve`) directly into the model's brain as single tokens.
- **Output:** Saves the compiled tokenizer to `models/custom-tokenizer/tokenizer.json`.

### Phase 3: Foundation Pre-Training (The Heavy Lift)
This is where the neural network actually learns to predict the next token based on the raw data.
- **Script:** `uv run python train/pretrain_custom_model.py`
- **What it does:** Initializes the 3B PyTorch architecture from scratch (randomized weights) and begins distributed training across your GPU cluster. It inherently uses `bfloat16` mixed precision for stability on modern GPUs (like RTX 40-series/Ada Lovelace).
- **Hardware Requirement:** Requires a cluster of heavy GPUs (e.g., 8x A100s or H100s) running for days or weeks.

### Phase 4: Alignment & Supervised Fine-Tuning (SFT)
Once the model understands Linux and cyber-security naturally, it must be aligned to act as an Autonomous SOC Analyst.
- **The Data:** You will export the `IncidentReport` and Human Review history from the MongoDB database (generated by the MERN Report Website).
- **The Process:** Teach the model: *"When you see this eBPF trace, output this exact JSON format with classification, MITRE tags, and action."*

### Phase 5: GGUF Export and Deployment
1. Export the final PyTorch weights to `.gguf` format (highly compressed and optimized for fast CPU/GPU inference).
2. Load the file into Ollama:
   ```bash
   echo "FROM ./agrus-3b-final.gguf" > Modelfile
   ollama create agrus-3b-engine -f Modelfile
   ```
3. Update the Python API `.env` file to point to your new proprietary model:
   ```bash
   SECURITY_AI_OLLAMA_MODEL=agrus-3b-engine
   ```

The `SecurityDecisionEngine` will now seamlessly route all enterprise telemetry through your custom brain!
